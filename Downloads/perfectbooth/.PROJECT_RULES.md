# Perfect Booth — Existing Next.js Project Rules

You are working inside an existing Next.js project for Perfect Booth.

The project already contains multiple completed sections on the Home Page.

Do not create a new Next.js project.
Do not rebuild the Home Page from scratch.
Do not delete or replace the existing completed sections.
Do not change the existing project structure unless a change is technically necessary.

Before implementing my request, inspect the current project and understand:

- The installed Next.js and React versions
- Whether the project uses App Router or Pages Router
- The current Home Page structure
- The existing Home Page sections
- The order of the sections
- Existing layout and container components
- Existing colors and design tokens
- Existing typography
- Existing button components
- Existing animation system
- Existing responsive breakpoints
- Existing localization system
- Existing Arabic RTL implementation
- Existing reusable UI components
- Existing image handling
- Existing package dependencies

Treat all completed Home Page sections as the approved visual reference for the rest of the website.

Any new section must look like it belongs to the same page and the same design system.

## Existing Work Protection

When implementing a new section:

- Preserve all existing sections.
- Preserve the current section order unless I explicitly request a change.
- Do not redesign unrelated sections.
- Do not replace working components unnecessarily.
- Do not overwrite the global stylesheet.
- Do not reset Tailwind configuration.
- Do not change the package manager.
- Do not install duplicate packages.
- Do not remove Arabic or English functionality.
- Do not break RTL layouts.
- Do not modify routes unrelated to the request.
- Do not rename existing components without a clear technical reason.
- Do not create duplicate Button, Container, Heading, or Section components if equivalent components already exist.

Only modify the files required for the requested section.

## Next.js Rules

Follow the architecture already used in the project.

Use:

- `next/image` for content images
- `next/link` for internal navigation
- TypeScript
- Existing path aliases
- Existing utilities
- Existing design tokens
- Existing localization utilities

Use Server Components by default if the project uses App Router.

Only add `"use client"` when the component genuinely requires:

- State
- Effects
- Event handlers
- Browser APIs
- Client-side animation
- Interactive controls

Do not turn the entire Home Page into a Client Component because one section needs interactivity.

Keep interactive behavior inside focused Client Components.

## Component Reuse

Before creating anything new, search the existing project for reusable components such as:

- Container
- SectionHeading
- Button
- ImageReveal
- ProjectCard
- ServiceCard
- AnimatedCounter
- Marquee
- CTA
- Typography components
- Motion wrappers

Reuse and extend existing components whenever possible.

Create a new component only when the requested section has genuinely different behavior or structure.

Every new Home Page section should be stored as a reusable component, following the current project conventions.

Example:

```tsx
<HomeHero />
<AboutSection />
<ServicesSection />
<ProcessSection />
<FeaturedProjectsSection />
<PartnersSection />
<CTASection />
```

Do not place a large section directly inside the main `page.tsx` file.

The page file should mainly compose the sections.

## Bilingual Requirements

Every new section must support both Arabic and English.

Requirements:

- English uses LTR.
- Arabic uses RTL.
- All visible content must come from the existing localization system.
- Do not hardcode English text inside reusable components.
- Do not create a second localization system.
- Add translations to the existing Arabic and English translation files.
- Use natural Arabic copy, not literal machine translation.
- Adjust Arabic line height and typography where needed.
- Adapt arrows and directional icons for RTL.
- Do not mirror project images.
- Make sure the layout remains visually balanced in both languages.

Use CSS logical properties where possible.

## Image Rules

Use real Perfect Booth project images whenever they are available in the project.

For every image:

- Use `next/image`.
- Define a controlled aspect ratio.
- Add responsive `sizes`.
- Prevent layout shifts.
- Provide localized alt text.
- Avoid stretching.
- Avoid low-quality placeholders.
- Lazy-load below-the-fold images.
- Prioritize only essential above-the-fold images.

Do not add unrelated generic stock images.

## Motion Rules

Follow the animation style already used in the completed sections.

Animations must remain:

- Subtle
- Premium
- Smooth
- Restrained
- Performance-conscious

Prefer:

- Fade-up
- Image reveal
- Staggered content
- Slow image scale
- Underline movement
- Subtle arrow movement

Avoid:

- Bouncing
- Fast rotations
- Excessive parallax
- Animating every element
- Long delays before content appears

Respect `prefers-reduced-motion`.

## Accessibility

Every new section must include:

- Semantic section markup
- Correct heading hierarchy
- Keyboard-accessible links and buttons
- Visible focus states
- Sufficient contrast
- Meaningful image alt text
- Reduced-motion support
- Accessible interactive elements

Do not use clickable `div` elements.

## Implementation Workflow

For every new request:

1. Inspect the current Home Page and related components.
2. Identify reusable existing components.
3. Identify the correct location for the new section.
4. Implement the section using the existing design system.
5. Add Arabic and English content.
6. Add RTL behavior.
7. Add responsive styling.
8. Add restrained animation only where useful.
9. Run TypeScript checks.
10. Run linting.
11. Fix all errors caused by the implementation.
12. Confirm that no existing section was broken.
